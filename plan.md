# Plan for the *Success and Luck* simulation

## Context

This project is an interactive visualization of the simulation model from the first appendix of Robert Frank's *Success and Luck: Good Fortune and the Myth of Meritocracy* (2016). 
The simulations show that in large competitive fields, the most successful contestant is rarely the most talented — instead, winners tend to be among the luckiest, even when luck accounts for only a tiny fraction of performance.

The main emphasis is on **visualization and interactivity**, since the simulation logic itself is simple. 
A basic Python reproduction exists in `simulation.py` for reference and validation.

This project also serves as a learning exercise in Next.js, TypeScript, and React for someone coming from Python.

## The model

Each simulated contest has N contestants. Each contestant has three scores drawn uniformly from [0, 100]:
- **Ability**
- **Effort**
- **Luck**

Performance is calculated as:

```
performance = (1 - luckWeight) * (ability + effort) / 2 + luckWeight * luck
```

where `luckWeight` is a parameter between 0 and 1 (typically 0.01 to 0.20).

The contestant with the highest performance score wins. Note that ability and effort are weighted equally and together make up the non-luck share.

Key outputs per simulation batch (M contests with the same parameters):
- Average luck score of winners
- Percentage of winners who did NOT have the highest ability + effort score
- Distribution of winner luck scores
- The gap between the most skilled contestant's performance and the actual winner's performance

### Why the results are striking

Among N contestants, many will be clustered near the top of the skill distribution.
Because `(ability + effort) / 2` averages two uniform draws, it follows a **triangular distribution** (peaked at 50, less variance than a single uniform draw), making the top contestants even more tightly bunched.
Luck then acts as a tiebreaker among near-equals.
The larger N is, the more contestants are packed near the ceiling, and the more decisive luck becomes — even at very small weights.

## Parameters and their ranges

| Parameter       | Variable     | Range            | Default | Notes                                    |
|-----------------|-------------|------------------|---------|------------------------------------------|
| Contestants     | `n`         | 100 – 100,000    | 1,000   | Log scale on slider                      |
| Luck weight     | `luckWeight`| 0.01 – 0.20      | 0.05    | Linear, displayed as percentage          |
| Contests        | `m`         | 100 – 10,000     | 1,000   | Number of contests per simulation batch  |

## Tech stack

- **Framework**: Next.js (App Router) with TypeScript
- **Styling**: Tailwind CSS
- **Charts**: Recharts (React-native charting, works well with Next.js)
- **Hosting**: Vercel (free tier, auto-deploys from GitHub on push)
- **Repo**: Public GitHub repo `skill-and-luck`

### Project structure

```
src/
  lib/
    simulation.ts       # Pure simulation logic, no UI dependencies
    types.ts            # Shared TypeScript types/interfaces
  app/
    page.tsx            # Main page: controls + visualizations
    layout.tsx          # Root layout (generated by create-next-app)
  components/
    Controls.tsx        # Sliders and run button
    ResultsSummary.tsx  # Key statistics display
    LuckChart.tsx       # Distribution of winner luck scores
    SkillVsLuck.tsx     # Scatter plot or similar: winner skill vs luck
    ContestTimeline.tsx # Animated contest playback (stretch)
```

## Build phases

### Phase 1: Scaffold and simulate
- Scaffold Next.js project with `create-next-app`
- Implement `simulation.ts` with the three-factor model
- Validate output against `simulation.py` reference values
- Create a minimal page that runs a simulation on button click and logs results to the page

**Done when**: you can click "Run" and see correct numbers rendered on the page.

### Phase 2: Controls and basic charts
- Build the controls panel with sliders for n, luckWeight, and m
- Add a results summary showing key stats (average winner luck, % where most skilled didn't win)
- Add a histogram of winner luck scores using Recharts
- Add a bar chart reproducing Frank's Figure A1.1 (average winner luck by parameters)

**Done when**: you can adjust parameters, run simulations, and see charts update.

### Phase 3: Richer interaction
- Scatter plot showing winners vs displaced champions: amber dots (lucky winners) vs blue dots (the most-skilled contestant they beat), plotted as skill score (x) vs luck score (y). Only contests where luck displaced skill are shown, so both clouds are the same size. The amber cloud sits higher (more luck); the blue cloud sits further right (more skill).
- Extended simulation data: `topSkillScore` and `topSkillLuck` added to `ContestWinner` to enable the displaced-champion dots.
- Responsive layout tweaks.

**Done when**: a non-technical user can explore the simulation and understand the result without reading Frank.

### Phase 4: Animation and polish (stretch)
- Animate a single contest: show contestants appearing, scores accumulating, winner emerging
- **Single-contest jitter/beeswarm view (discussed, deferred from Phase 3):** Show all N contestants from one contest as dots along a single axis (skill score or performance), with random y-jitter to separate overlapping points. Circle or highlight the winner and the most-skilled contestant separately. This gives a visceral "zoom in" on one race that complements the aggregate scatter. Works best as part of the step-through/animation feature — run one contest at a time and watch the winner emerge. In Recharts this requires manually adding a random `jitterY` field to each point (computed once and stored, so it doesn't re-randomise on re-render).
- "Step through" mode: run one contest at a time and watch results accumulate
- Add an explanatory text panel that narrates what the user is seeing
- Visual polish: transitions, color palette, typography
- Consider adding the two-factor (skill + luck) model as a toggle for comparison

## Design notes

- The page should work as a single scrollable page, not multi-page navigation
- Controls at the top or in a sidebar, results below
- Color should encode luck vs skill visually (e.g., a consistent blue for skill, amber/gold for luck)
- Large contest counts (n > 10,000) may be slow in the browser — consider running simulation in a Web Worker if it blocks the UI
- All computation is client-side; there is no backend or API

## Validation

Compare against these reference values from `simulation.py` (two-factor model, for rough validation):
- n=100, m=100, w=0.05 → average winner skill ≈ 98.5, average winner luck ≈ 73
- As n increases, winner luck should increase and "most skilled wins" percentage should decrease
- With w=0, the most skilled contestant should always win (sanity check)
- With w=1, winner luck should average around the expected max of a uniform sample: 100·(n/(n+1))
